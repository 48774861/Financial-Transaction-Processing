# Real-Time Financial Transaction Data Processing for Fraud Detection Systems
Andrew Keuhn, Spencer Neill, Marc Pham  
## Abstract
Modern financial systems process massive volumes of transaction data in real time.  Fast fraud detection is essential, but many systems are limited by a critical bottleneck at the data preprocessing stage before analysis can begin. In this work, we present a GPU-accelerated preprocessing pipeline using CUDA that continuously partitions large-scale transaction data into batches, efficiently sorts each batch using a parallel radix sort, and organizes all transactions by user ID and timestamp using a parallel merge. Leveraging the data-parallel nature of these operations significantly improves our system’s execution time compared to CPU-based solutions like Apache Spark, achieving a speedup of 15.54. Individually, the parallel radix sort and parallel merge have speedups of 28.06 and 35.22 respectively. This work highlights the importance of optimizing preprocessing as a key step in building scalable, real-time financial analytics systems.

## Introduction
   Modern financial platforms like banks and fintech applications must process millions of transactions per minute before fraud detection models can even begin their work. The core challenge is not just detecting fraud, but doing so quickly enough for banks to take preventative action in real time. To achieve this, systems must preprocess and organize transaction data at high speed so that machine learning models and downstream analytics can run without delay. This preprocessing stage includes sorting transactions by user and timestamp, merging multiple incoming data streams, and structuring the data for efficient analysis. Optimizing this step is critical, because faster data readiness leads directly to quicker fraud detection, reduced financial risk, and better scalability as transaction volumes continue to grow.  
   
   In this project, we developed a high-performance data processing pipeline using CUDA and GPUs. The transaction dataset was first divided into smaller, independent batches of about one million transactions each, reflecting real-world ingestion rates. To improve memory efficiency on the GPU, we stored the data in a Struct-of-Arrays (SoA) format, where each attribute (such as timestamp or user ID) is kept in its own separate array. Each batch was then sorted by user ID and timestamp using a parallel radix sort, which processes data digit by digit and is well-suited for parallel execution. After sorting the batches independently, we applied a parallel merge to combine them into a single, globally sorted dataset. Our objective was to outperform CPU-based systems like Apache Spark (an industry-standard tool for large-scale data processing) by leveraging GPU parallelism to further accelerate transaction preprocessing.  
   
   The use of parallel computing is justified by the inherently data-parallel nature of the problem. Radix sorts and merging can be decomposed into independent operations across large datasets, making them ideal for GPU execution. While CPU-based systems like Apache Spark are already highly optimized and perform very well at large-scale data processing, their performance is still limited by the number of CPU cores and general-purpose hardware constraints. Moving these operations to GPUs using CUDA can significantly increase our throughput by taking advantage of thousands of lightweight threads running in parallel. In financial systems where decisions must be made in milliseconds, even minor speed improvements can significantly enhance the performance and responsiveness of downstream processes like fraud detection.  

## Literature Survey
   AI in Financial Services: Real-Time Fraud Detection on Cloud-Native GPU Clusters by Venkata Karunakar Uppalapati presents a full fraud detection pipeline that processes transaction data from ingestion to final prediction. The system relies on technologies such as Apache Kafka for handling high-throughput data streams, Redis for fast access to recent transactions, and Apache Iceberg for managing large-scale historical data. These components work together to clean and enrich incoming transactions before feeding them into a two-stage model, where a fast initial filter screens most transactions and more advanced models like gradient boosting and graph-based methods analyze the remaining suspicious cases. The system achieves high throughput and low latency largely by using GPU acceleration, specifically hardware like the NVIDIA A100 Tensor Core GPU, to speed up model inference.   

   However, the primary emphasis of this work is on improving the speed and performance of the fraud detection models rather than optimizing the earlier data preparation stages. While steps like data cleaning, sorting, and structuring are included in the pipeline, they are not explored in depth or optimized using specialized algorithms. This is a limitation because these preprocessing steps are critical for efficiently handling large-scale transaction data, especially when transactions must be ordered by attributes such as user ID and timestamp. As a result, although the system performs very well in terms of prediction speed and accuracy, it may still experience bottlenecks in the preprocessing phase when organizing large and complex datasets.   

   Real-Time Financial Fraud Detection: An Intelligent Data-Driven Framework Integrating Machine Learning, Stream Processing, and Big Data Analytics for High-Velocity Transaction Monitoring by Farah Arzu and collaborators presents a comprehensive real-time fraud detection system. Using streaming data tools like Apache Kafka and Apache Spark, the framework  processes large volumes of transaction data as they are generated, allowing continuous monitoring instead of delayed batch analysis. The system then applies a mix of machine learning approaches like Random Forest, Gradient Boosting, and Autoencoders to detect both known fraud patterns and unusual or previously unseen behavior. In addition, the system includes online learning for models to adapt over time as well as explainable AI techniques to make predictions more transparent, which is important for financial regulations and user trust. Overall, the framework is effective at combining scalability, adaptability, and accuracy, achieving high throughput and low decision latency in fast-moving financial environments.   

   However, similar to Uppalapati’s work, this approach focuses more on improving overall fraud detection performance and real-time system design rather than optimizing data preprocessing. Although it supports real-time data ingestion and feature extraction, it does not deeply address efficient operations like sorting, merging, or organizing large-scale transaction data, which can become bottlenecks. It also does not rely heavily on GPU acceleration for speeding up its models, focusing instead on distributed streaming and cloud infrastructure. As a result, while the system achieves strong real-time detection performance and scalability, it may still be limited in preprocessing efficiency and does not fully leverage hardware acceleration for either data processing or model computation.   

   The Real-time Suspicious Detection Framework for Financial Data Streams proposed by Gadimov and Birihanu builds an end-to-end streaming architecture for analyzing financial transactions. It integrates a container-based distributed infrastructure with machine learning models such as autoencoders, using reconstruction error to identify anomalous behavior. A major strength of this work is its strong focus on real-time processing, enabling transactions to be evaluated as they arrive rather than relying on offline batch analysis, which improves responsiveness and makes it well-suited for fast-moving financial environments. Another advantage is its scalable system design, which leverages distributed components to handle high-volume data streams effectively. However, the approach places most of its emphasis on the streaming infrastructure and detection model, while earlier stages of the pipeline, such as data organization and preprocessing, are treated in a relatively standard way, leaving less focus on optimizing the efficiency of data preparation itself.   

## Commercial Impact
Across modern fraud detection systems, the choice of machine learning model – whether it is gradient boosting, neural networks, or graph-based approaches – ultimately depends on the availability of well-structured data at the right time. This makes the preprocessing stage a critical dependency in the overall pipeline. By using dedicated GPUs to accelerate transaction sorting, merging, and formatting, data can be made available to these models significantly faster than with CPU-based systems. As a result, model inference can begin earlier and produce fraud detection predictions with reduced end-to-end latency.   

From a commercial perspective, this reduction in delay has direct operational value for financial institutions. Faster predictions enable earlier detection of suspicious activity, allowing banks and fintech platforms to intervene before fraudulent transactions are fully completed or propagated. Even minimal improvements in preprocessing speed can scale across millions of transactions, leading to meaningful reductions in financial exposure and fraud-related losses. Ultimately, GPU-accelerated preprocessing strengthens the entire fraud detection pipeline by improving responsiveness, scalability, and risk mitigation regardless of the underlying machine learning approach.   

## Difference from Prior Works
Our work shifts the focus away from model inference and instead optimizes the data preprocessing layer of fraud detection machine learning systems. Specifically, we address scalable sorting, merging, and data layout transformations required before fraud detection can begin, emphasizing parallel algorithm design to efficiently organize transactions by keys such as the transaction date and the client sending the transaction. This enables deterministic grouping and chronological ordering at scale, which is often underexplored in existing literature despite being essential for downstream behavioral analysis. The advantage of our approach is that it directly reduces preprocessing latency and improves data locality, leading to faster end-to-end pipeline performance regardless of the downstream model choice. Additionally, by focusing on GPU-friendly primitives, our system better exploits hardware parallelism for structured data movement rather than only for machine learning inference.   

Our work does not address the full fraud detection stack like a majority of previous literature do and purely focuses on incorporating parallelism into the transaction data preprocessing. Unlike Uppalapati’s system, we do not incorporate adaptive feature engineering, model orchestration, or real-time inference optimization. His work excels in intelligent, scalable fraud detection using advanced models combined with cloud-native infrastructure for end-to-end system efficiency. Unlike the system proposed by Arzu and collaborators, our work does not focus on end-to-end real-time fraud detection or on building full machine learning pipelines. Their work was strong at detecting both known and emerging fraud patterns and supporting adaptive learning and explainability for regulatory compliance. Unlike Gadimov and Birihanu’s system, our approach does not involve containerized distributed streaming architectures or autoencoder-based real-time anomaly detection pipelines, which are designed for continuous transaction monitoring. Their work created a responsive processing pipeline that analyzes transactions as they arrive rather in offline batches.   

Our approach complements and strengthens these systems by improving the efficiency and scalability of the foundational data preprocessing stage that everything else depends on. By deeply optimizing low-level operations such as sorting and organizing large-scale transaction datasets, we make it easier for analysts and downstream models to work with clean, structured, and well-ordered data. Ultimately, regardless of which machine learning model or system is used, accelerating preprocessing through GPU-based parallelism has the potential to make fraud detection pipelines faster, more efficient, and more scalable.   

## Detailed Methodology
In this project, we designed and implemented a high-performance transaction preprocessing pipeline using CUDA and GPUs to accelerate large-scale financial data processing. We used the Financial Transactions Dataset: Analytics from Kaggle (created by Caixabank Tech), which contains 13.3 million transaction records (1.26 GB), and partitioned it into independent batches of about one million transactions to reflect real-world ingestion rates. To maximize memory coalescing, we stored the data in a Struct-of-Arrays (SoA) format, where each attribute (transaction date and client ID) is placed in its own contiguous array to improve memory coalescing. Each batch was then sorted by (client_id, timestamp) using a parallel radix sort, followed by a parallel merge to combine the batches into a single globally sorted dataset. Our baseline for comparison consisted of CPU-based radix sort and merge implementations used in industry-standard tools like Apache Spark.   

### Radix Sort Kernel Methodology
To initially prepare the dataset for the GPU, transaction records are read and processed in batches of up to 1,000,000 rows from a CSV file. In a production real-time fraud detection system, this step would instead be implemented as a continuous ingestion pipeline, similar to a micro-batching architecture in Apache Kafka, where incoming transactions are buffered over short, fixed time windows rather than stored in static files. During ingestion, each transaction is parsed and transformed into a uniform data structure with the full string of the original CSV row and a compact 64-bit sort key, ensuring consistency across multiple heterogeneous data sources.   
As part of this normalization stage, each event is assigned a compact 64-bit composite sort key designed specifically for efficient batch-level organization during preprocessing. The transaction timestamp is first converted into epoch seconds, and the Client ID is converted into an integer. These two values are then packed into a single sortable integer where the upper 32 bits represent the timestamp and the lower 32 bits represent the Client ID. This structure allows all events to be ordered using a single 64-bit integer comparison across multiple asynchronous streams, with the timestamp being the primary sort criteria and the client ID being the secondary sort criteria. It also provides flexibility in experimental design, since modifying sort priority can be achieved by swapping the bit allocation between fields without changing the underlying computational complexity. Because this stage consists primarily of parsing, normalization, and key construction, its computational cost is treated as a common overhead across both approaches and is not the primary differentiator in performance evaluation.   

The GPU stage of the pipeline reorders each preprocessed batch of transaction data into a consistent and globally comparable structure before it is passed to downstream fraud detection systems. This step uses CUB DeviceRadixSort through its key-value interface (SortPairs), which is specifically designed for high-performance sorting of fixed-width integer keys. Radix sort is a natural fit for this workload because it avoids direct comparisons between elements. Instead, it processes the 64-bit keys in small bitwise groups, sorting the data in multiple passes. This approach produces effectively linear-time performance with respect to batch size, making it significantly more efficient than traditional O(N log N) comparison-based sorting when working with large datasets. Additionally, on the GPU, each pass is executed in parallel across thousands of threads, allowing the algorithm to fully exploit data-level parallelism. The use of uniform fixed-width keys reduces branching differences between threads, which minimizes warp divergence. In addition, memory accesses are highly structured and coalesced, since the GPU operates on one contiguous array of 64-bit sort keys and one parallel array of 32-bit indices. CUDA Unbound (CUB) further optimizes this process through carefully tuned shared memory usage and multi-pass execution strategies, ensuring stable performance across different batch sizes.   

A key design decision in this architecture is that GPU sorting is performed at the batch level, enabling pipeline parallelism across system components. While the GPU processes and reorders one batch, the CPU is free to continue ingesting and preparing subsequent batches of transactions. This overlap decouples the high-throughput but compute-intensive reordering step from the continuous ingestion pipeline, improving overall system throughput and utilization.   

The use of the SortPairs interface is also critical because it performs a stable key-value sort, where the 64-bit composite keys define ordering while a parallel array of indices preserves references to the original transactions. As a result, the GPU produces not only a sorted sequence of keys but also a corresponding permutation of event indices that can be transferred back to the CPU. This avoids the need to move full transaction payloads during sorting, significantly reducing PCIe transfer overhead and keeping the GPU stage focused purely on high-throughput reordering.   

### Merge Kernel Methodology
After the GPU radix sort stage, the pipeline must merge sorted segments into a single globally ordered output array. This is done using a GPU-parallel merge strategy designed to eliminate the sequential bottleneck of traditional merge operations while maintaining correctness across large, pre-sorted datasets.   

The key idea is to avoid merging sequentially from the start of the arrays. Instead, the algorithm partitions the output space and assigns each GPU thread a specific contiguous range of final output positions. Each thread is responsible for producing only the elements that fall within its assigned segment. To do this correctly, the system must determine how many elements from each of the two input sorted arrays contribute to any given output position.   

This is handled using a merge-path style partitioning step. For a given output index, the algorithm computes a split point that divides the two input arrays such that the total number of selected elements matches that output position while preserving sorted order. This split is found using a binary search over valid partition positions in one of the arrays, repeatedly checking boundary conditions between the two arrays to ensure that all elements to the left of the partition are smaller than or equal to those on the right. This transforms the problem from element-by-element merging into a boundary-finding problem that can be solved efficiently in logarithmic time.   

Once the correct partition boundaries are determined, each thread performs a standard linear merge within its assigned subranges. It uses a two-pointer scan over the corresponding slices of the two input arrays, writing the smallest available element at each step into the correct position in the output array. Because each thread works on a disjoint segment of the output, there are no write conflicts and no need for synchronization between threads.   

Overall, this design ensures that work is evenly distributed across the GPU by assigning each thread a similar number of output elements. It also preserves the correctness of the global ordering produced by the radix sort stage while enabling the merge operation to scale efficiently across many parallel threads.   

After the GPU completes the sorting and merging stages, the resulting ordered indices are transferred back to the CPU, where they are used to reconstruct the final globally ordered transaction list. The CPU then maps these indices back to the original event records and writes the fully ordered dataset to a CSV file for downstream analysis and evaluation. The final reconstruction and output step is identical in both the CPU-only and GPU-accelerated implementations, so it does not introduce a meaningful performance difference between the two approaches.   

## Concluding Remarks
In this work, we showed that the preprocessing stage of financial data pipelines, which is often treated as a secondary concern, can be a major source of performance improvement when carefully optimized. By leveraging CUDA and GPU parallelism, we built a high-performance pipeline that accelerates large-scale transaction sorting and merging using parallel radix sort and parallel merge, using a Struct-of-Arrays memory layout to maximize memory coalescing. These design choices allowed us to efficiently map data-parallel operations onto thousands of GPU threads, significantly improving throughput compared to CPU-based systems like Apache Spark. More importantly, our work demonstrates that optimizing how data is organized before model inference can be just as impactful as optimizing the models themselves. Our results show that the full preprocessing pipeline processed the 1.26GB financial transaction dataset in approximately 43 seconds, sustaining an average throughput of 307,185,352 transactions per second and delivering an overall speedup of 15.54. These results highlight how GPU parallelism can substantially reduce preprocessing time and make large-scale transaction data ready for analysis much faster than traditional CPU-based approaches.   

These improvements indicate that large-scale transaction data can be prepared quickly enough to support real-time or near real-time fraud detection systems. By reducing system latency and enabling faster decision-making, GPU-accelerated preprocessing can remove critical bottlenecks in existing pipelines and help fraud systems scale more effectively with growing data volumes. Through this project, we gained practical insight into GPU architecture, memory access patterns, and parallel algorithm design, as well as the challenges of balancing computation and data movement. Overall, our findings highlight that focusing on foundational data operations is a powerful and complementary strategy for improving the performance of end-to-end machine learning systems.   

Future work will focus on integrating the preprocessing pipeline directly into a downstream machine learning model to measure true end-to-end system speedup and better evaluate real-world impact. From an implementation perspective, a key improvement is incorporating CUDA streams to overlap data transfer with kernel execution, thereby reducing idle time and increasing throughput. Another important extension is scaling the pipeline across multiple GPUs through data partitioning and efficient inter-GPU communication. Together, these steps would move the system closer to production-scale deployment while providing a more complete understanding of its performance benefits.
